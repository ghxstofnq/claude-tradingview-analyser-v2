## Why there was no second try

The 10:11 MES short was early and failed. Trace review confirmed why no second executable attempt surfaced:

1. The current deterministic walker has a **session primary-trade latch**: once any walker reaches `packet_ready`, the session is marked as primary-taken.
2. A stop outcome does **not** reset that latch.
3. Later candidates did appear, but they were blocked as `session_primary_already_taken`:
   - 11:19 ET: later Inversion long candidate blocked.
   - 11:54 ET: later Inversion short candidate blocked.
4. No second clean short appeared before the 10:41 ET move into the original TP1 area.

This is a useful future design question, not an oracle-promotion fact. If we want retry-after-early-stop behavior later, it should be implemented as a separate controlled rule with risk limits, not hidden inside this label.
